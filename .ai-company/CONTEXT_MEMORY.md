# CONTEXT & MEMORY — 上下文、事实与长期记忆治理

## 1. 目标

建立“项目知道什么、为什么这样决定、哪些信息仍未知”的可靠记忆，避免每次任务重新猜测，也避免旧信息长期污染决策。

---

## 2. 信息层级

### L1：即时任务上下文

仅对当前任务有效：临时日志、草稿、实验结果、短期假设。

### L2：项目运行知识

跨任务有效：架构、业务规则、模块所有权、环境、发布流程、常见故障。

### L3：组织制度与战略

长期有效：公司目标、批准权限、数据政策、核心技术原则和不可妥协规则。

不同层级不可混淆。临时实验结论不得未经批准升级为长期规则。

---

## 3. 单一事实来源

建议来源：

| 信息 | 首选事实来源 |
|---|---|
| 业务目标 | 项目章程 / 已批准 PRD |
| 当前实现 | 代码仓库与部署制品 |
| 数据结构 | 实际 schema / migration |
| API 契约 | OpenAPI/契约测试/实现 |
| 权限 | 权限策略、服务端实现、测试 |
| 架构决策 | ADR |
| 任务状态 | 任务记录/看板 |
| 发布状态 | CI/CD 与部署平台 |
| 运行状态 | 监控、日志、健康检查 |

README 描述与实际实现冲突时，不自动假设任何一方正确，应建立修复项。

---

## 4. 记忆写入规则

写入长期记忆前必须满足：

- 信息具有跨任务价值。
- 来源可追踪。
- 已区分事实、决定、偏好和假设。
- 不包含秘密、令牌、密码或不必要个人数据。
- 有所有者或更新机制。

建议格式：

```markdown
- ID:
- Type: fact | decision | rule | preference | risk | lesson
- Statement:
- Source:
- Owner:
- Valid from:
- Review date:
- Supersedes:
```

---

## 5. 过期与冲突

- 所有会变化的信息应有复审日期。
- 新决策覆盖旧决策时，旧记录保留但标记 `superseded`。
- 冲突信息不得静默合并。
- 发现信息过期时，先阻止其继续作为依据，再安排更新。

---

## 6. 上下文最小化

Agent 只读取完成任务所需的上下文：

- 减少无关信息带来的错误关联。
- 降低敏感数据暴露。
- 降低长上下文中规则被忽略的风险。

但不得为了节省上下文而省略关键约束、历史决策和安全规则。

---

## 7. 项目地图

项目接管后维护：

- 业务域和核心流程。
- 主要模块与负责人。
- 外部系统和供应商。
- 数据存储和敏感数据位置。
- 环境、部署和监控。
- 关键 ADR 与已知技术债。
- 高风险操作和运行手册。

项目地图用于导航，不替代具体实现和证据。

---

## 8. 任务结束后的知识沉淀

每个任务关闭时评估是否需要更新：

- README / 架构文档。
- ADR。
- 业务规则。
- 测试策略。
- 运行手册。
- 风险登记。
- 常见问题和故障知识库。
- 本制度规则。

只有具有可复用价值的内容进入长期记忆，避免积累大量无用日志。
